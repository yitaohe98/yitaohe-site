# Product Requirements Document (PRD)
# yitaohe.com
# Version 1.0 (MVP)

---

## 1. Product Positioning

### 1.1 Vision

A Chinese-first technical blog serving as both:

- A long-term technical knowledge platform
- A personal brand landing page

The site prioritizes clarity, sustainability, and extensibility.

English support and engagement features are future enhancements and are not part of the MVP scope.

---

## 2. Product Goals

### 2.1 MVP Goals

1. Launch a Chinese-first technical blog
2. Provide a professional Overview (personal brand) page
3. Implement structured blog organization
4. Enable static generation with automated deployment
5. Provide basic SEO support
6. Use a local Markdown-based authoring workflow

---

### 2.2 Out of Scope (MVP)

The following features are intentionally excluded from v1.0:

- Multi-language (English) content
- Views / Likes metrics
- Comment system
- Search functionality
- User accounts
- Backend database

These may be added in future versions.

---

## 3. Technical Architecture

### 3.1 Tech Stack

- Static Site Generator: Hugo
- Hosting & Deployment: Cloudflare Pages
- Content Format: Markdown
- Version Control: GitHub

---

### 3.2 Architectural Principles

1. Static-first architecture
2. No runtime backend dependencies
3. All content stored in Git repository
4. Clean separation between content and infrastructure
5. Extensible without over-engineering

---

## 4. Information Architecture

### 4.1 Top-Level Navigation

- Overview
- Blogs
- Comments (placeholder page)

---

### 4.2 URL Structure

```
/overview/
/blogs/
/blogs/{category}/{slug}/
/comments/
```

---

### 4.3 Blog Categories

Categories are used for internal organization and filtering.

Example categories (extensible):

- system-design
- backend
- investing
- life
- notes

Categories are not exposed as top-level navigation items.

---

## 5. Content Model Specification

All content must be written in Markdown.

### 5.1 Front Matter Specification

Each post must include:

```yaml
title: "Distributed Rate Limiter Design"
date: 2026-02-24
description: "Step-by-step breakdown of distributed rate limiting systems."
primary_category: "system-design"
categories: ["system-design"]
tags: ["redis", "distributed-systems"]
draft: false
```

---

### 5.2 Rules

- primary_category is required.
- Slugs must be English.
- Multiple categories are allowed.
- Tags are flexible and optional.
- Content must remain pure Markdown.

---

## 6. Page Specifications

### 6.1 Overview Page

Must include:

1. Personal introduction
2. Technical focus statement
3. Featured posts (3–5 curated articles)
4. Contact information (GitHub / LinkedIn / Email)

Goal:
A visitor should understand who you are and what you specialize in within 10 seconds.

---

### 6.2 Blogs Listing Page

Features:

- Reverse chronological order
- Category-based browsing
- Tag filtering
- Pagination support

---

### 6.3 Blog Post Page

Must support:

- Title
- Table of Contents (for long-form posts)
- Code syntax highlighting
- Previous/Next navigation (optional)
- SEO meta tags

---

### 6.4 Comments Page (MVP Placeholder)

For v1.0:

- Static page
- Provide contact method
- No interactive comment system

---

## 7. Content Authoring Workflow

### 7.1 Current Workflow (MVP)

1. Write Markdown in VSCode
2. Preview locally using `hugo server`
3. `git push`
4. Cloudflare Pages auto-build and deploy

---

### 7.2 Design Principles

- Content must remain Markdown-based
- No database dependency
- No runtime API dependency
- Fully version-controlled
- Easily diffable and reversible

---

### 7.3 Migration Readiness

Future support for:

- Git-based CMS
- Multi-language content

Preconditions:

- Content structure remains Markdown-based
- Directory structure remains stable
- No content locked inside proprietary platforms

---

## 8. SEO (MVP Scope)

The site must support:

- Auto-generated sitemap.xml
- RSS feed
- Meta descriptions
- Open Graph metadata
- Clean, readable URLs

SEO Objective:
Technical articles should be discoverable via search engines.

---

## 9. Future Enhancements (Good to Have)

### 9.1 Multi-language (EN)

Future design:

- Pre-configured language structure
- Language switch placeholder in header
- English content added incrementally

---

### 9.2 Views / Likes

Future goal:

- Display total views and likes per article
- No user identity tracking

Implementation direction:

- Serverless functions
- Managed storage
- No impact on static architecture

---

### 9.3 Comment System

Possible future options:

- Giscus
- Serverless API

---

## 10. Non-Functional Requirements

- Lighthouse score ≥ 95
- First contentful paint < 1s
- No runtime backend
- Global CDN distribution
- High stability and low maintenance cost

---

## 11. Success Metrics

- 1–3 posts per week
- ≥ 20 posts within 3 months
- Sustainable long-term maintenance
