# Product Requirements Document (PRD)
# yitaohe.com
# Version 1.0 (MVP)

---

## 1. Product Positioning

### 1.1 Vision

A technical blog and personal brand site (EN and 中文) serving as:

- A long-term technical knowledge platform
- A personal brand landing page

The site prioritizes clarity, sustainability, and extensibility. Content is edited primarily via a Git-based CMS (Decap CMS); the site remains static-first and fully version-controlled.

---

## 2. Product Goals

### 2.1 MVP Goals

1. Launch a technical blog with structured organization (categories, tags)
2. Provide a professional Overview (personal brand) page
3. Support multi-language content (EN and 中文) with a clear URL and editing model
4. Enable static generation with automated deployment (Cloudflare Pages)
5. Provide basic SEO support (sitemap, RSS, meta, clean URLs)
6. Use a **Git-based CMS (Decap CMS)** as the primary way to create, edit, and delete posts

---

### 2.2 Out of Scope (MVP)

The following are not part of the current scope:

- Views / Likes metrics
- Interactive comment system (static contact only)
- Full-text search
- User accounts or backend database

These may be addressed in future versions.

## 3. Technical Architecture

### 3.1 Tech Stack

- **Static Site Generator:** Hugo
- **Hosting & Deployment:** Cloudflare Pages
- **Content format:** Markdown (stored in repo under `content/`)
- **Primary editing UI:** Decap CMS (Git-based CMS at `/admin/`)
- **Version control:** GitHub (content and code in the same repository)

---

### 3.2 Architectural Principles

1. **Static-first** — All pages generated at build time; no runtime backend.
2. **Content in Git** — All posts and pages are Markdown files in the repository; Decap CMS writes to the same repo.
3. **Clean separation** — Content in `content/`, layouts in `layouts/`, assets in `static/`; docs in `docs/`.
4. **Extensible** — Add categories, tags, or languages without changing core architecture.

## 5. Content Model Specification

All content must be written in Markdown.

### 5.1 Front Matter Specification

Each post must include:

```yaml
title: "Distributed Rate Limiter Design"
date: 2026-02-24
description: "Step-by-step breakdown of distributed rate limiting systems."
primary_category: "system-design"
categories: ["system-design"]
tags: ["redis", "distributed-systems"]
draft: false
```

For multi-language pairing, also set `translationKey` to a stable id shared by the same post in other languages (e.g. `translationKey: "distributed-rate-limiter"`). Optional: `featured: true` for overview featured section.

---

### 5.2 Rules

- `primary_category` is required.
- Slugs must be English.
- Multiple categories are allowed.
- Tags are flexible and optional.
- Content must remain pure Markdown (body supports text, code blocks, and images; see [Blog design](blog-design.md) §11).

## 7. Content Authoring Workflow

The **primary** way to create, edit, and delete blog posts is **Decap CMS** at the site’s `/admin/` path. Content is stored as Markdown in the Git repository; Hugo builds the site from that content. Alternative: edit Markdown directly in the repo (e.g. in an editor or on GitHub) and push — the result is the same.

### 7.1 Creating a post

1. Open the site’s admin UI (e.g. `https://www.yitaohe.com/admin/`).
2. Log in with GitHub (requires write access to the repository).
3. Choose the blog collection for the desired language (e.g. “Blog (EN)” or “Blog (中文)”).
4. Click **New [Blog post]** (or equivalent).
5. Fill in the form:
   - **Title**, **Date**, **Description**
   - **Primary category** (required), **Categories** (list), **Tags** (list)
   - **Draft** (yes/no), **Featured** (yes/no) if supported
   - **Body** (Markdown; use the editor for text, code blocks, and image uploads)
6. For images: upload in the editor; they are stored in the post’s folder (page bundle) or a configured media folder.
7. Click **Save** (or **Publish**). Decap commits to the repository (e.g. to the default branch).
8. Cloudflare Pages (or your host) rebuilds on push; the new or updated post appears on the site after the build completes.

Slug and path are derived from the CMS config (e.g. `{category}/{slug}`) so URLs follow the structure `/blogs/{category}/{slug}/`.

### 7.2 Editing a post

1. Open `/admin/` and log in with GitHub.
2. In the blog collection, open the existing post from the list.
3. Change title, description, body, categories, tags, draft status, etc.
4. Click **Save**. Decap creates a new commit; the site rebuilds and the updated content is live after deploy.

### 7.3 Deleting a post

1. Open `/admin/` and log in with GitHub.
2. In the blog collection, open the post to delete.
3. Use the **Delete** (or equivalent) action in the CMS. Decap removes the file from the repository and commits the change.
4. The site rebuilds; the post disappears from the listing and its URL returns 404 (or redirect, if configured).

If the CMS does not expose delete: remove the post’s file (e.g. `content/en/blogs/{category}/{slug}/index.md`) and its folder in the repo via Git, then push; the result is the same.

**Alternative:** Authors can create, edit, or delete posts by changing Markdown files directly in the repository (e.g. in an editor or on GitHub) and pushing. The site builds from the same content; Decap CMS is the preferred UI, not the only path.

### 7.4 Design principles

- **Markdown-based** — All content is stored as Markdown; no database or proprietary format.
- **Version-controlled** — Every create/edit/delete is a Git commit; history is auditable and reversible.
- **No runtime API** — The CMS talks to GitHub; the deployed site is static and has no backend dependency.
- **Migration-friendly** — Content and structure stay in the repo; switching editors or tooling does not lock you in.
